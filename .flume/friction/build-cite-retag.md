**Symptom:** a repo-wide `sed` swap of `specs/architecture/15-kinds.md` → `specs/model/representation.md` (and siblings) silently rewrote test *fixture* literals in `src/kind.rs` (`source_path: PathBuf::from("specs/architecture/15-kinds.md")`, a `write_surface(...)` arg, a `+++` TOML fixture string) that happened to share the exact retired path string. The fixtures exist to prove the extractor derives a `source_dir` placement fact from a path's parent directory; the swap silently flipped the expected value from `"architecture"` to `"model"`, desyncing them from a hardcoded `assert_eq!(..., Some("architecture"))` two call sites away. `cargo build` stayed green (it's just a string), and the break was only caught by manually re-deriving the fixture's intent and cross-checking assertions — an automated `cargo test` run would also have caught it, but only if run *before* trusting the sed as "done."

**Cost this tick:** ~20 minutes of manual tracing (grep for suspicious `Some("word")` placement assertions across `src`+`tests`) to find and fix 4 corrupted literals across one file, on top of the fan-out already needed for the citation sweep itself.

**Suggested fix:** for any repo-wide mechanical string replace targeting doc-comment citations, always follow with a full `cargo test` (not just `cargo build`) before considering the pass done — build success proves nothing about fixture-string correctness. Better: scope the sed to lines matching a comment-marker prefix (`///`, `//!`, `//`) rather than a bare string match, so it can't reach into string literals/fixture data at all.
